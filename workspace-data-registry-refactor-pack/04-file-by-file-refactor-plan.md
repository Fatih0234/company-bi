# 04 — File-by-File Refactor Plan

This is the main implementation map. Follow it slowly. Do not treat it as optional guidance.

The goal is to replace the default TLC/MinIO data path with a generic workspace-local data registry while preserving the content-only workspace architecture.

## Global implementation order

1. Add registry/data-source specs and tests.
2. Add a registry refresh implementation that scans workspace `data/`.
3. Generate shadow runtime Evidence source files under `sources/files/`.
4. Make `cmux-evidence validate` work with registered files.
5. Expose registered tables through `duckdb-bi`.
6. Inject registered table context through `evidence-context.ts`.
7. Update dashboard/data-discovery skills.
8. Update README/demo docs.
9. Demote TLC/MinIO to optional examples.

Do not start with step 9.

---

## Current role

Main Python CLI for workspace lifecycle:

- create new workspaces,
- open CMUX layout,
- validate,
- diff,
- publish,
- manage content-only shadow runtime.

## Functions/areas to inspect first

Search within the file for:

- `ensure_shadow_runtime`
- `ensure_content_shadow_runtime`
- `new_content_analysis`
- `write_content_snapshot`
- `validate`
- `diff`
- `publish`
- `main` / command dispatch
- any helper that loads `.cmux/evidence.json` or `.cmux/workspace.json`

## Planned changes

Add workspace data registry support and CLI commands.

### Add constants

Add constants near existing workspace constants:

```python
DATA_REGISTRY_PATH = ".cmux/data-registry.json"
DATA_PROFILE_PATH = ".cmux/data-profile.json"
GENERATED_FILES_SOURCE_NAME = "files"
SUPPORTED_WORKSPACE_DATA_EXTENSIONS = {".csv", ".tsv", ".parquet", ".json", ".jsonl"}
```

Do not add `.xlsx` yet.

### Add helper functions

Add pure helper functions where possible:

```python
workspace_data_dir(workspace_root: Path) -> Path
workspace_data_registry_path(workspace_root: Path) -> Path
workspace_data_profile_path(workspace_root: Path) -> Path
load_data_registry(workspace_root: Path) -> dict[str, Any]
save_data_registry(workspace_root: Path, registry: dict[str, Any]) -> None
slugify_table_alias(filename_or_stem: str) -> str
unique_alias(base_alias: str, used_aliases: set[str]) -> str
fingerprint_file(path: Path) -> str
scan_workspace_data_files(workspace_root: Path) -> list[dict[str, Any]]
refresh_workspace_data_registry(workspace_root: Path, *, profile: bool = False) -> dict[str, Any]
generate_workspace_file_sources(workspace_root: Path, shadow_root: Path, registry: dict[str, Any]) -> None
```

Keep the helpers deterministic and easy to test.

### Registry refresh behavior

`refresh_workspace_data_registry` should:

1. Ensure `data/` exists.
2. Scan `data/` recursively up to a reasonable depth, for example 5.
3. Include only supported extensions.
4. Ignore hidden files, temp files, `.DS_Store`, and directories starting with `.`.
5. Create stable aliases.
6. Preserve aliases from an existing registry when the same path still exists.
7. Mark deleted files as `missing` or remove them according to the spec. MVP recommendation: keep missing entries with `status: "missing"` so the user can see stale references.
8. Write `.cmux/data-registry.json`.
9. Do not perform expensive full profiling unless requested.

### Source generation behavior

`generate_workspace_file_sources` should:

1. Ensure `shadow_root/sources/files/` exists.
2. Write a generated `connection.yaml` for an in-memory DuckDB source if needed.
3. Remove old generated `.sql` files that are no longer in the registry.
4. Write one generated SQL file per ready table.
5. Use `workspace-data/<relative-file>` paths.
6. Include a clear generated header comment.
7. Never write generated source SQL to the content workspace.

Example generated file:

```sql
-- Generated by cmux-evidence data refresh. Do not edit by hand.
-- Source file: data/orders.csv
select *
from read_csv_auto('workspace-data/orders.csv')
```

### Modify `ensure_shadow_runtime`

Current behavior links workspace data as `workspace-data`; preserve and strengthen that.

Required changes:

- Keep linking `pages`, `queries`, `reports` from workspace to shadow runtime.
- Keep linking workspace `data` as `workspace-data`.
- Stop treating root `data` as default business data for new generic flow.
- After linking `workspace-data`, load/refresh registry if it exists or if data files exist.
- Generate `sources/files/*.sql` from registry.
- Do not require files to exist for empty workspaces.

Be careful: this function is used to repair missing shadow runtime. It must be idempotent.

### Add `data` subcommands

Add command dispatch for:

```bash
./bin/cmux-evidence data list
./bin/cmux-evidence data refresh
```

MVP only. Do not implement `add`, `remove`, or `profile` unless MVP is complete.

`data refresh` should work from:

- main project root if targeting current/root workspace where applicable,
- a content-only workspace root.

Preferred behavior from content workspace:

```bash
cmux-evidence data refresh
```

should:

1. load current workspace metadata,
2. refresh registry,
3. ensure/repair shadow runtime,
4. generate sources,
5. optionally run `npm run sources -- --sources files` from the shadow runtime if files exist,
6. print registered tables.

If no files exist:

```text
No workspace data files found.
Add CSV, TSV, Parquet, JSON, or JSONL files under data/ and run data refresh again.
```

### Do not break

- `new`
- `open`
- `validate`
- `diff`
- `publish`
- content-only/non-git behavior
- legacy worktree fallback behavior

## Acceptance criteria for `bin/cmux-evidence`

- New workspaces still create successfully.
- Empty `data/` does not crash.
- `data refresh` creates `.cmux/data-registry.json`.
- Shadow runtime gets generated `sources/files/*.sql`.
- Existing tests still pass.
- New tests cover registry behavior.

# 4. New `scripts/ensure_workspace_sources.sh`

## Purpose

Generic source bootstrap used by Evidence dev/build flows.

## Responsibilities

1. Detect whether running from a content workspace or shadow runtime.
2. Locate workspace root and shadow runtime root from `.cmux/workspace.json` / `.cmux/evidence.json`.
3. Ensure workspace `data/` exists.
4. If data files exist, ensure `.cmux/data-registry.json` exists.
5. Generate `sources/files/*.sql` from registry.
6. Run `npm run sources -- --sources files` only if there are registered ready tables.
7. Exit successfully if no files exist.

## Implementation recommendation

This script can be a thin shell wrapper around Python functions in `bin/cmux-evidence` or a new Python helper module. Avoid duplicating registry logic in bash.

Preferred:

```bash
#!/usr/bin/env bash
set -euo pipefail
python3 scripts/workspace_data_registry.py ensure-sources
```

or:

```bash
./bin/cmux-evidence data refresh --no-open --from-dev-script
```

Be careful to avoid recursion if `data refresh` itself runs Evidence commands.

# 7. New `pi-pkg/extensions/duckdb-bi/src/lib/workspace-data-registry.ts`

## Purpose

Read and normalize `.cmux/data-registry.json` for Pi tools.

## Responsibilities

- Load registry safely.
- Validate minimal shape.
- Return registered tables.
- Convert entries into public tool output.
- Distinguish ready/missing/error/stale tables.
- Do not modify registry unless called by an explicit refresh/register tool.

## Suggested API

```ts
export type WorkspaceDataTable = {
  alias: string;
  qualifiedName: string;
  path: string;
  format: string;
  status: "ready" | "missing" | "error" | "stale";
  columns?: Array<{ name: string; type?: string; semanticRole?: string }>;
  rowCount?: number;
  warnings?: string[];
};

export function loadWorkspaceDataRegistry(config: DuckDbBiConfig): WorkspaceDataRegistry | undefined;
export function registeredTablesForContext(config: DuckDbBiConfig): WorkspaceDataTable[];
```

## Acceptance criteria

- Handles missing registry gracefully.
- Handles malformed registry by returning warnings, not crashing the whole session.
- Does not expose raw absolute paths in normal public output.

# 8. `pi-pkg/extensions/duckdb-bi/src/tools/data-sources.ts`

## Current role

Registers `duckdb_data_sources`, listing Evidence sources, local data files, exports, reports, and runtime dirs.

## Planned changes

Add registered workspace tables to the response.

New response shape should include:

```json
{
  "registered_tables": [
    {
      "name": "files.orders",
      "alias": "orders",
      "path": "data/orders.csv",
      "format": "csv",
      "status": "ready",
      "recommended_for_dashboard": true
    }
  ],
  "files": [
    {
      "path": "data/raw_export.csv",
      "alias": "raw_export",
      "recommended_for_dashboard": false,
      "registration_hint": "Run cmux-evidence data refresh to register this file."
    }
  ]
}
```

Existing `evidence_sources` should remain, but registered tables should appear before raw files in the response.

## Important behavior

- If a file is registered, do not also present it as an unregistered raw file unless useful with `already_registered: true`.
- Registered tables should be the preferred dashboard source.
- Raw discovered files are fallback/proposed data sources.

## Acceptance criteria

- `duckdb_data_sources` shows registered tables after refresh.
- Registered tables are marked `recommended_for_dashboard: true`.
- Unregistered files are marked `recommended_for_dashboard: false` with a registration hint.

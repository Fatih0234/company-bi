# 03 — Non-Negotiable Invariants

This is the most important file in the pack. The implementation agent must read it before making any code changes.

If any proposed change violates one of these invariants, stop and redesign the change.

## 1. Do not break content-only workspaces

The content-only workspace model is central to this repo.

A user-facing analysis workspace must remain clean and focused on content:

```text
pages/**
queries/**
reports/**
data/**
.cmux/** metadata
.pi/** symlinked Pi resources and local scratch
```

The user-facing workspace must not expose or require direct editing of root runtime implementation files:

```text
package.json
package-lock.json
bin/**
scripts/**
sources/**
node_modules/**
.evidence/**
```

The shadow runtime may contain runtime files and generated Evidence source files. The content workspace should not.

## 2. Do not make page SQL use raw file paths

Evidence dashboard pages must query stable registered source names such as:

```sql
from files.orders
from files.customers
```

Do not write dashboard page SQL like this:

```sql
from read_csv_auto('data/orders.csv')
from read_parquet('data/orders.parquet')
from read_json_auto('data/events.jsonl')
```

Raw file reads may be used internally by generated source SQL or safe DuckDB exploration tools, but not in user-facing Evidence page queries.

Reason: page SQL should remain portable, readable, and stable even if file paths are regenerated or shadow runtime paths differ.

## 3. Do not publish raw workspace data by default

The current privacy model must be preserved.

Default publish may include:

```text
pages/report.md
queries/**
```

Default publish must not include:

```text
data/**
.pi/duckdb/**
.cmux/data-profile.json
raw original uploads
local scratch files
```

If later publish modes include data, they must be opt-in and explicit. Do not implement data publishing accidentally.

## 4. Do not remove TLC/MinIO first

Do not begin this refactor by deleting existing TLC/MinIO files.

Correct sequence:

1. Build generic workspace data registration.
2. Prove it with tests.
3. Prove Evidence can build from `files.<alias>`.
4. Update Pi context/skills.
5. Then demote TLC/MinIO to an optional example or remove from the default path.

Deleting the old path first will make debugging harder and could break current tests or runtime assumptions.

## 5. Do not manually edit secrets or connection files

Never read, print, copy, generate, or expose real secrets.

Do not edit:

```text
.env*
**/connection.yaml
**/connection.options.yaml
```

Exception: this refactor may generate a non-secret `connection.yaml` under the shadow runtime's generated `sources/files/` folder if required by Evidence. That generated file must not contain secrets.

## 6. Keep generated files clearly generated

Every generated source SQL file should include a header comment:

```sql
-- Generated by cmux-evidence data refresh. Do not edit by hand.
```

Generated files belong in the shadow runtime, not the user-facing workspace.

Generated registry/profile files should be deterministic where possible. Avoid timestamps in files that are compared in tests unless timestamps are part of the explicit schema.

## 7. Preserve stable aliases

The same file path should keep the same alias across refreshes.

Example:

```text
data/orders.csv -> files.orders
```

If another file conflicts, add a suffix to the new file:

```text
data/orders.csv -> files.orders
data/archive/orders.csv -> files.orders_2
```

Do not randomly rename aliases on every refresh. Unstable aliases will break report pages.

## 8. Do not widen filesystem access casually

All file discovery, profiling, registration, generated SQL, exports, and reports must stay inside approved workspace/runtime roots.

Do not allow:

```text
../outside.csv
/Users/someone/secret.csv
~/.ssh/id_rsa
.env.local
```

The CLI may support `data add /external/file.csv` later, but it must copy the file into workspace `data/` before registration. The registry should point to workspace-relative paths.

## 9. Keep DuckDB tools safe

`duckdb-bi` is intentionally a safe BI tool layer, not a generic shell escape hatch.

Keep these properties:

- readonly SQL by default,
- destructive SQL blocked,
- one safe query execution path,
- no shell-string concatenation,
- outputs under `.pi/duckdb/**`,
- query audit logging,
- path validation.

Do not add a tool that runs arbitrary shell commands to inspect data.

## 10. Do not break Pi package loading

This repo depends on `pi-pkg/package.json` to load project-local Pi resources.

If you create any new Pi asset, register it under the correct section:

```json
{
  "pi": {
    "extensions": [],
    "skills": [],
    "prompts": [],
    "themes": []
  }
}
```

If you only modify an existing registered extension or skill, do not add duplicate entries.

## 11. Do not over-scope Excel in MVP

MVP file formats:

- CSV
- TSV
- Parquet
- JSON
- JSONL

Do not implement native Excel/XLSX in the first pass unless all MVP behavior is complete and tested. Excel should be a later phase that converts sheets to registered CSV/Parquet extracts.

## 12. Empty workspaces must remain valid

A brand-new workspace with no files in `data/` must still:

- open in CMUX,
- run the dev server,
- show useful onboarding text,
- not crash `cmux-evidence data refresh`,
- not crash validation because data is absent.

The absence of data should produce a helpful message, not a failure.

## 13. Do not make the registry the only source of truth for user files

The actual files in `data/` are the physical source of truth.

The registry is a derived control file that records stable aliases, paths, schema, and status.

If the registry and `data/` disagree, refresh should detect it and mark missing/stale files.

## 14. Preserve current commands

Do not break existing commands:

```bash
./bin/cmux-evidence new
./bin/cmux-evidence open
./bin/cmux-evidence list
./bin/cmux-evidence current
./bin/cmux-evidence status
./bin/cmux-evidence validate
./bin/cmux-evidence diff
./bin/cmux-evidence publish
./bin/lumen-pi
```

New `data` commands must be additive.

## 15. Preserve current tests before adding new behavior

Run existing tests before and after meaningful changes.

If an existing test fails, either:

- fix the implementation while preserving intended behavior, or
- update the test only when the product behavior intentionally changed and the new expected behavior is documented in this pack.

Never ignore failing tests.

## 16. Keep docs and prompts aligned with code

If you change the data workflow, update all relevant docs/prompts/skills:

- `README.md`
- `pi-pkg/skills/evidence-dashboard/SKILL.md`
- `pi-pkg/skills/data-discovery/SKILL.md`
- `pi-pkg/extensions/duckdb-bi/README.md`
- any prompt that mentions TLC/MinIO/source names

Do not leave old TLC-first instructions in the default workflow after the generic path is implemented.

## 17. Prefer one small phase at a time

This is a large refactor. Implement one phase, validate, then continue.

Recommended order:

1. registry schema/tests,
2. registry refresh,
3. generated shadow source files,
4. Evidence validation,
5. DuckDB/Pi context integration,
6. skill/prompt updates,
7. docs/demo updates,
8. TLC demotion.

Do not implement all phases in one uncontrolled patch.

## 18. Report back clearly

After each phase, report:

- files changed,
- tests run,
- commands run,
- behavior verified,
- known gaps,
- next recommended phase.

If blocked, stop and report the exact blocker. Do not make broad speculative edits.

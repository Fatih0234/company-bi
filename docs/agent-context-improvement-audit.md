# Agent Context Improvement Audit

## 1. Executive summary

Company BI already has a sophisticated Pi-specific context stack: project-local Pi assets in `pi-pkg/`, dynamic per-turn Evidence context in `pi-pkg/extensions/evidence-context.ts`, content-workspace metadata under `.cmux/`, generated `.cmux/pi-context.md`, CMUX layout wiring in `bin/cmux-evidence`, and workflow skills for dashboard building, review, BI thinking, data discovery, browser preview, and publishing.

The biggest opportunity is to make the same operating model durable and discoverable for agents that do not enter through Pi. Today, the root `AGENTS.md` only covers Pi asset registration. A generic coding agent opened at the repository root learns almost nothing about content-only workspaces, Evidence query rules, safe edit boundaries, dynamic context layers, docs routing, privacy rules, or the distinction between root app development and dashboard authoring. Generated content workspaces also do not receive an `AGENTS.md`; they depend on Pi dynamic injection, `.cmux/pi-context.md`, starter pages, and `.pi` symlinks.

Recommended target: keep dynamic context small and stateful, keep skills task-specific, and add two compact durable instruction layers:

1. A root `AGENTS.md` that explains repo modes, safe edit policy, Pi registration, docs routing, and context layer map.
2. A generated workspace-level `AGENTS.md` or equivalent that gives non-Pi agents enough guidance for content-only dashboard work without copying the full Pi prompt stack.

Add snapshot tests for generated agent context, generated workspace instructions, and Pi package registration. Do not solve this by pasting all skills and dynamic context into `AGENTS.md`; that would create stale, bloated, duplicated instructions.

Confirmed facts:

- `bin/lumen-pi` disables global Pi resources and loads `pi-pkg`.
- `bin/pi-full` loads `pi-pkg` without disabling global resources.
- `bin/cmux-evidence` creates content-only workspaces with `pages/`, `queries/`, `data/`, `reports/`, `.cmux/workspace.json`, `.cmux/evidence.json`, `.cmux/pi-context.md`, and `.pi` symlinks.
- `evidence-context.ts` injects dynamic context through `before_agent_start` and truncates at 12000 characters.
- The dynamic context intentionally avoids `.env*` and `**/connection.yaml`.
- Current tests assert that content-workspace context uses content-only terms and an absolute runtime helper path.

Inferences:

- Generic agents opened at root receive `AGENTS.md` but do not automatically receive Pi dynamic context or skills.
- Generic agents opened in generated content workspaces receive no durable `AGENTS.md` unless their tool runtime independently discovers `.cmux/pi-context.md` or page templates.
- Pi dashboard sessions may be less deterministic than intended when launched through `pi-full` because global resources are not disabled.

Unknowns:

- Whether CMUX or Pi has an additional external mechanism that automatically reads `.cmux/pi-context.md` beyond what is visible in this repository.
- Whether `pi-full` is intentionally used for generated workspaces to preserve global user resources, or whether this drifted from the isolated `lumen-pi` design.
- Whether generated workspace `.pi` symlinks are required for current Pi discovery when `-e "$PROJECT_ROOT/pi-pkg"` is already passed by the launcher.

## 3. Agent startup journeys

### Codex or generic coding agent at repo root

1. Starts in `/Volumes/T7/projects/company-bi`.
2. Reads root `AGENTS.md` if the agent supports it.
3. Learns only the Pi asset registration rule from `AGENTS.md`.
4. Must manually inspect `README.md`, `.cmux/evidence.json`, `bin/cmux-evidence`, `pi-pkg/`, and `.agent/docs/` to discover the true architecture.
5. Does not automatically receive Pi skills, prompt templates, dynamic context, or slash commands.

Risk: root development, Pi asset work, Evidence authoring, and content-workspace privacy rules are not distinguished for non-Pi agents.

### Pi launched through `bin/lumen-pi`

1. Launcher computes `PROJECT_ROOT`.
2. Session directory is workspace-local under `.cmux/pi-sessions` when launched inside a content workspace with `.cmux/workspace.json`; otherwise under root `.cmux/pi-sessions`.
3. Runs `pi --no-extensions --no-skills --no-prompt-templates --no-themes -e "$PROJECT_ROOT/pi-pkg"`.
4. Pi loads only project-local package resources plus normal settings/provider/auth.
5. `evidence-context.ts` injects dynamic Evidence context before each agent start.
6. Skills and prompts are available from `pi-pkg/package.json`.

Strength: isolated and reproducible. Gap: generated content workspace config currently points at `pi-full`, not `lumen-pi`.

### Pi launched in a content-only workspace

1. CMUX layout starts the agent terminal with cwd equal to the content workspace root.
2. `bin/cmux-evidence` generated `.pi/` symlinks to root `pi-pkg` subdirectories and settings/package files.
3. Workspace metadata lives in `.cmux/workspace.json`; helper/runtime config lives in `.cmux/evidence.json`.
4. Pi dynamic context finds the Evidence root by walking up to `.cmux/evidence.json`.
5. Dynamic context uses the content workspace for editable pages/data and the runtime root for source catalog/docs helper paths.
6. `.cmux/pi-context.md` exists as a static fallback snapshot.

Risk: non-Pi agents launched directly in this directory do not get a standard durable instruction file.

### CMUX-created Evidence workspace

1. `bin/cmux-evidence new "Title"` dispatches to content-only mode when `.cmux/evidence.json` has `workspaceMode: content-only`.
2. It creates a clean content workspace under the configured workspace directory, defaulting to `~/.local/share/lumen-bi/workspaces/<projectId>/`.
3. It creates a shadow runtime under the configured runtime directory, defaulting to `~/.local/share/lumen-bi/runtime/<projectId>/`.
4. The content workspace owns `pages/`, `queries/`, `data/`, `reports/`, and `.cmux/`.
5. The shadow runtime owns the generated Evidence app, symlinks content dirs, generated `sources/`, package files, scripts, node modules, and cache.
6. The CMUX layout uses the content workspace for the Pi pane and the shadow runtime for the Evidence dev pane.

Risk: no generated `AGENTS.md`; context is split across metadata, `.cmux/pi-context.md`, starter pages, and Pi dynamic injection.

### Dashboard-building session

1. Pi receives dynamic context with workspace roots, page paths, data registry state, source catalog, required dashboard workflow, data access rules, docs routing, and CMUX preview rules.
2. `evidence-dashboard` skill provides the full build workflow.
3. `evidence-bi-thinking` skill defines the Insight Candidate Scan and Report Design Plan.
4. `data-discovery` skill plus `duckdb-bi` tools provide profiling and auditable queries.
5. `evidence-quality-guard` blocks unsafe or unvalidated page writes in Pi.
6. `evidence-health-check` and CMUX/browser skills support preview validation.

Risk: the workflow is strong inside Pi but weakly expressed for external agents.

### Publishing session

1. `cmux-evidence publish` validates first.
2. For content-only workspaces, confirmation says publishing shares `pages/report.md` and `queries/**`; draft notes and local data stay private by default.
3. It materializes publishable content into a review worktree branch under `analysis/<slug>`.
4. It warns about local data files when `publishData` is not set.
5. It creates a commit and tries to push/open a PR through `gh`.

Risk: privacy-sensitive publishing rules are not in root `AGENTS.md` and not in a generated workspace-level agent instruction file.

## 4. Gaps and risks

### Context missing for generic agents

- `AGENTS.md` only covers Pi asset registration.
- It does not explain root app development vs content-only dashboard authoring.
- It does not tell agents to avoid `.env*`, `connection.yaml`, local data exposure, generated shadow runtime edits, or raw file-path SQL in Evidence pages.
- It does not route agents to `.agent/docs/evidence-oss/ROUTES.md`, `.agent/docs/pi/ROUTES.md`, or `.agent/docs/cmux-com/ROUTES.md`.

### Context hidden inside dynamic injection only

- `evidence-context.ts` contains critical rules: two SQL contexts, registered source names, docs lookup, CMUX non-disruptive behavior, required BI thinking workflow, and secret file avoidance.
- Generic agents do not receive that hook.
- `.cmux/pi-context.md` exists, but the main skill labels it legacy/fallback and it can go stale.

### Duplicated instructions

Similar rules appear in several places:

- Data access rules: `README.md`, `evidence-context.ts`, `evidence-dashboard/SKILL.md`, `data-discovery/SKILL.md`.
- Report workflow: `evidence-dashboard/SKILL.md`, `evidence-dashboard.md` prompt, `evidence-bi-thinking/SKILL.md`, `evidence-quality-guard/README.md`, dynamic context.
- Docs lookup rule: `evidence-context.ts`, `evidence-dashboard/SKILL.md`, `evidence-dashboard.md`, `evidence-quality-guard/README.md`.
- Safe edit policy: `.cmux/evidence.json`, generated `.cmux/pi-context.md`, `evidence-dashboard/SKILL.md`.

Duplication is not inherently bad, but there is no single manifest explaining which layer is authoritative.

### Stale or machine-local config risk

- Root `.cmux/evidence.json` contains absolute local paths under `/Volumes/T7/projects/company-bi`.
- Dynamic context points to a source docs path under `/Users/fatihkarahan/.opensrc/...`.
- `.agent/repos/pi/INDEX.md` and other indexed docs include local opensrc cache paths.
- `.cmux/pi-context.md` is a generated snapshot and may not match updated dynamic context or `.cmux/evidence.json`.
- Some docs still describe `.pi/extensions/...` as the source location even though project assets live in `pi-pkg/` and are symlinked into content workspaces.

### Over-broad context risk

- `evidence-context.ts` can inject up to 12000 characters every turn.
- The dynamic context includes stable rules, runtime state, source catalog, docs routing, data registry, and quality workflow together.
- Skills and prompt templates duplicate many of the same stable rules.

This can crowd the model context and make updates hard to reason about. The right split is durable stable rules in files, dynamic state in injection, and deep workflows in skills.

### Secret/privacy risks

Positive controls:

- `.cmux/evidence.json` blocks `.env*`, `**/connection.yaml`, `.github/**`, package files, `bin/**`, scripts, and `.pi/extensions/**` in content workspaces.
- `evidence-context.ts` states that secrets and connection files are not read.
- `ensure_workspace_sources.sh` says it must not read secrets or publish data.
- Publishing keeps draft notes and local data private by default.

Remaining risks:

- Generic agents do not get these privacy rules from root or workspace `AGENTS.md`.
- Generated source `connection.yaml` exists in shadow/runtime source directories after data refresh; blocked path rules mention it, but generic agents may inspect it unless instructed.
- Local data under `data/` is allowed for analysis but must not be published or quoted wholesale; this distinction is not in root `AGENTS.md`.

### Validation gaps

Existing tests cover:

- Content workspace creation, split roots, shadow repair, context command output, diff/publish behavior.
- Data registry creation/source generation.
- Data build behavior.
- Some extension-level tests under `pi-pkg/extensions/...`.

Missing tests:

- Root `AGENTS.md` contains required sections and stays compact.
- Generated content workspaces include durable agent instructions.
- Generated workspace instructions match `.cmux/evidence.json` edit policy.
- Pi package registration covers every asset directory/file under `pi-pkg/extensions`, `pi-pkg/skills`, `pi-pkg/prompts`, and `pi-pkg/themes` that should be loadable.
- Dynamic Evidence context has a snapshot test for core sections and a negative test for secret/connection file references.
- `bin/lumen-pi`, `bin/pi-full`, root `.cmux/evidence.json`, and generated workspace config agree on intended isolated vs global resource behavior.

## 5. Recommended target architecture

### Layer 1: root `AGENTS.md`

Purpose: durable, high-signal instructions for generic agents and root repository development.

Should contain:

- One-paragraph product/workspace model.
- Task-mode matrix: root development, Pi asset development, dashboard authoring, data registry/DuckDB, CMUX/browser preview, publishing/privacy.
- Safe read/edit rules, including secret/data restrictions.
- Instruction to prefer metadata in `.cmux/workspace.json` and `.cmux/evidence.json` over hard-coded paths.
- Pi asset registration rule currently present.
- Docs routing pointers to `.agent/docs/evidence-oss/ROUTES.md`, `.agent/docs/pi/ROUTES.md`, `.agent/docs/cmux-com/ROUTES.md`.
- Rule that generated content workspaces are user content; shadow runtime is generated and should not be edited directly.
- Validation command map.

Should not contain:

- Full Evidence component docs.
- Full Pi skill bodies.
- Full dynamic source catalogs.
- Large workflow templates copied from prompts.
- Machine-local absolute paths.

Minimal root `AGENTS.md` outline:

```markdown
# AGENTS.md

## Project Model
- Company BI / LUMEN is a local-first Evidence BI workspace system.
- Root repo owns runtime code, Pi assets, CMUX helpers, docs indexes, and tests.
- Content-only workspaces own analysis content: pages/, queries/, reports/, data/, .cmux/.
- Shadow runtimes are generated Evidence apps; do not edit them directly.

## Task Modes
- Root development: edit bin/, scripts/, pi-pkg/, tests/, docs as requested.
- Pi asset development: register new assets in pi-pkg/package.json.
- Dashboard authoring: prefer pages/, queries/, reports/ in the content workspace.
- Data work: use registered tables files.<alias>; refresh registry before page SQL.
- CMUX preview: inspect browser/dev server after substantial dashboard edits.
- Publishing/privacy: publish report and queries by default; do not publish data/drafts unless explicitly requested.

## Safety
- Do not read or expose .env*, **/connection.yaml, credentials, tokens, or private local data.
- Do not edit package/runtime/source files from a content-only dashboard task unless explicitly asked.
- Prefer .cmux/workspace.json and .cmux/evidence.json over hard-coded paths.

## Docs Routing
- Evidence OSS components: .agent/docs/evidence-oss/ROUTES.md.
- Pi extensions/skills/packages/settings: .agent/docs/pi/ROUTES.md.
- CMUX workspace/browser automation: .agent/docs/cmux-com/ROUTES.md.

## Pi Asset Registration
- Existing registration rule...

## Validation
- Root tests: python3 -m pytest tests
- Pi asset tests: npm test in the asset directory where available
- Dashboard validation: <workspace-helper> validate, preview inspection
```

### Layer 2: generated workspace-level `AGENTS.md`

Purpose: make content-only workspaces usable by Codex, Cursor, Claude Code, and other agents that do not run Pi.

Recommended path: `AGENTS.md` at the content workspace root.

Why not only `.cmux/pi-context.md`: `AGENTS.md` is a standard agent bootstrap file. `.cmux/pi-context.md` is useful but hidden, nonstandard, and already described as legacy/fallback by the Pi skill.

Should contain:

- This is a content-only analysis workspace.
- Editable paths: `pages/**`, `queries/**`, `reports/**`, `data/**`, approved `.cmux` metadata.
- Do-not-edit paths: runtime root, shadow runtime, package files, scripts, generated sources, `.env*`, `connection.yaml`.
- Page roles: `pages/index.md` brief, `pages/draft.md` sandbox, `pages/report.md` publishable report.
- Data rule: page SQL must use registered source names like `files.orders`; do not use raw file readers in Evidence page SQL.
- Where to find metadata and helper command.
- Privacy/publish rule.
- Validation commands.

Should not contain:

- Full dynamic data registry dumps.
- Long BI workflow templates.
- Full Evidence docs.
- Large analysis intention contents beyond a pointer to `.cmux/workspace.json`.

Generated workspace `AGENTS.md` outline:

```markdown
# AGENTS.md

## Workspace Type
This is a LUMEN content-only Evidence analysis workspace.
Edit analysis content here; runtime code lives elsewhere.

## Read First
- .cmux/workspace.json: title, goal/intention, page paths, preview URL, helper roots.
- .cmux/evidence.json: commands, port, safe edit policy.
- pages/index.md: human-readable brief.

## Edit Boundaries
- Allowed: pages/**, queries/**, reports/**, data/**, .cmux/workspace.json when updating the brief/status.
- Ask first: generated sources, components, runtime-root files.
- Do not read/edit: .env*, **/connection.yaml, credentials, tokens, shadow runtime internals.

## Page Roles
- pages/index.md: brief and workspace map.
- pages/draft.md: private exploration and plan.
- pages/report.md: polished publishable report.

## Data Rules
- Run data refresh after adding files to data/.
- Evidence page SQL must use registered source names like files.<alias>.
- Do not use read_csv_auto(), read_parquet(), read_json_auto(), or raw paths in page SQL.

## Validation
- Use the helper command from .cmux/evidence.json or .cmux/workspace.json.
- Run validate before declaring completion.
- Inspect preview when browser/CMUX is available.

## Privacy
Publishing shares report and queries by default. Drafts, local data, scratch files, credentials, and runtime files stay private unless explicitly requested.
```

### Layer 3: dynamic context injection

Purpose: current state that changes per workspace or turn.

Should contain:

- Current roots, title, slug, port, URL, CMUX IDs.
- Current intention summary.
- Current source catalog and data registry summary.
- Current data-profile snippets.
- Helper command path.
- Short reminders for rules that depend on workspace state.

Should not contain:

- Full static workflow manuals.
- Full component docs.
- Large source SQL previews when not needed.
- Machine-local docs paths unless unavoidable and clearly a cache location.

Recommendation: keep the `MAX_CONTEXT_CHARS` cap, but split stable rules into named compact sections and test the rendered output. Consider moving the stable docs-routing prose into a durable manifest and injecting only a pointer plus the two most important rules.

### Layer 4: Pi skills

Purpose: task-specific workflows and detailed procedural guidance.

Should contain:

- `evidence-dashboard`: dashboard build workflow, page conventions, validation sequence.
- `evidence-dashboard-review`: read-only review rubric.
- `evidence-bi-thinking`: analytical moves and report planning.
- `data-discovery`: data profiling workflow and query provenance.
- CMUX skills: browser/workspace/diagnostics details.

Should not contain:

- Constant copies of root architecture that should live in `AGENTS.md`.
- Machine-local absolute paths.
- Runtime state that belongs in dynamic context.

### Layer 5: prompt templates

Purpose: optional high-level mode presets for Pi sessions.

Should contain:

- Persona/task framing and compact quality bar.
- Short reference to skills and docs.

Should not contain:

- Complete duplicates of skills.
- Rules that must be enforced across all agents.

### Layer 6: docs routing/index files

Purpose: source-of-truth navigation for external tools and docs.

Should contain:

- Task routing, local index paths, upstream provenance.
- Read-only policy for external cached source.

Should not contain:

- Agent behavior policy that belongs in `AGENTS.md`.
- Secrets or connection examples with real credentials.

### Layer 7: validation/snapshot tests

Purpose: prevent invisible regressions in agent context.

Should include:

- Generated workspace instruction snapshot.
- Dynamic context snapshot for a fixture workspace.
- Context negative tests for secret file names/content.
- Manifest registration check for `pi-pkg`.
- Launcher/config consistency check.
- Docs route existence check.

## 7. Prioritized roadmap

### Quick wins

1. Expand root `AGENTS.md` using the minimal outline above while preserving the existing Pi asset registration rule.
2. Add a generated content-workspace `AGENTS.md` with compact edit/data/privacy/validation rules.
3. Add tests that assert generated workspaces contain that file and that it reflects configured safe edit policy.
4. Add a `docs/agent-context-manifest.md` that maps durable files, dynamic context, skills, prompts, and commands.
5. Fix stale docs that describe `pi-pkg` assets as if their source lives in `.pi/`.

### Medium improvements

1. Add a registration contract test for `pi-pkg/package.json`.
2. Add dynamic context snapshot or section-presence tests.
3. Decide and document whether generated CMUX Pi sessions should use isolated `lumen-pi` or non-isolated `pi-full`.
4. Replace machine-local docs paths in injected context with repo-relative route files plus optional cache hints.
5. Split stable guidance from dynamic context so per-turn injection is smaller.

### Deep architecture changes

1. Introduce a single agent context manifest that can generate root and workspace agent instructions from shared templates.
2. Add context-size budgets by layer: root AGENTS under about 150 lines, workspace AGENTS under about 120 lines, dynamic context under a tested cap, skills unrestricted but task-specific.
3. Add a full context harness that creates a fixture content workspace, refreshes data, renders dynamic context, and validates that Pi/generic agents would receive the expected minimum context.
4. Add policy checks that block publishing or PR creation when generated context files are missing or stale.

## 8. Acceptance criteria

Agent context is better when:

- A generic agent opened at repo root can identify the project model, task modes, safe edit boundaries, docs routes, and validation commands from `AGENTS.md` alone.
- A generic agent opened in a generated content workspace can safely build or review an Evidence report after reading workspace `AGENTS.md`, `.cmux/workspace.json`, and `pages/index.md`.
- Pi still receives dynamic state before each turn, but stable rules are not repeated in full across dynamic context, skill, and prompt.
- New Pi assets cannot be added without registration test failures unless explicitly marked private/experimental.
- Generated workspace instructions are covered by tests and update when edit policy config changes.
- Context files do not contain secrets, credentials, or real local data values.
- Publishing/privacy rules are visible to both Pi and non-Pi agents.
- Validation commands are explicit and mode-specific.
- Absolute local paths are limited to runtime metadata where needed and not embedded as authoritative docs paths.

## 10. Open questions

1. Should generated CMUX dashboard Pi panes use isolated `bin/lumen-pi` or non-isolated `bin/pi-full`?
2. Should generated content workspaces get root-level `AGENTS.md`, `.agents.md`, `.codex/`, `.cursor/rules/`, or only `AGENTS.md` plus `.cmux/pi-context.md`?

4. Should `AGENTS.md` include a short dashboard workflow, or only route agents to `pages/index.md`, `.cmux/workspace.json`, and docs?

7. Which Pi assets under `pi-pkg/` are intentionally unregistered or experimental?
8. Should `.cmux/pi-context.md` remain a legacy fallback, or become the canonical generated workspace context manifest?
9. Should dynamic context include docs cache absolute paths, or only repo-relative route files and upstream links?

## Do not do this

- Do not paste the full contents of Pi skills into root `AGENTS.md`.
- Do not paste Evidence component docs into agent instructions.
- Do not put dynamic data registry tables, source SQL previews, or data profiles into durable `AGENTS.md`.
- Do not rely only on Pi hooks for safety-critical privacy instructions.
- Do not make page templates the only source of workspace instructions.
- Do not expose `.env*`, `connection.yaml`, credentials, local data samples, or private source details in context files.
- Do not make generic agents edit shadow runtime files to fix dashboard content.
- Do not add context files without tests that prove generated workspaces receive them.

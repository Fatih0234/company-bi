# Success Criteria: SwiftRide Market Entry Playbook — Full Analysis + Slide Deck

## 1. Intended Outcome

**Primary outcome:** A complete, data-driven analysis answering all 7 SwiftRide questions, culminating in a self-contained HTML slide presentation that Maya Chen (VP of Operations) can use to make deployment decisions.

**Explicit requirements:**
- All 7 questions (Q1–Q7) must be answered with data-backed findings in dedicated Evidence pages
- A polished `pages/report.md` synthesizing all findings into a coherent narrative
- A self-contained HTML slide deck at `reports/swiftride-nyc-playbook/slides/index.html` following the `evidence-slides` skill pattern
- The deck must follow "Decision Memo" or "Executive Answer" pattern from `slide-story-patterns.md`
- Every recommendation must trace to specific queries/charts in the Evidence report
- The deck must include risk/caveat sections and an alternative "Plan B" scenario

**Inferred requirements:**
- SQL queries must be valid Evidence queries (validated via `duckdb_validate_evidence_sql`)
- Charts must use Evidence-native components (BarChart, LineChart, DataTable, BigValue)
- Zone joins must use `taxi_zone_lookup` for human-readable names
- Data cleaning: filter negative fares, handle null passenger counts, exclude extreme outliers (>100mi trips)
- The 7 analysis pages (q1–q7) should contain working queries and charts, not just placeholders

## 2. Functional Acceptance Criteria

### Analysis Pages (q1–q7)
- [ ] Q1 Zone Strategy: Top zones ranked by opportunity score (volume × revenue × gap), with zone names from lookup
- [ ] Q2 Time Strategy: Hourly and day-of-week demand curves with revenue overlay
- [ ] Q3 Competitive Positioning: Yellow vs green zone dominance map, gap analysis
- [ ] Q4 Trip Profile Strategy: Trip distance vs revenue scatter/histogram, sweet spot identification
- [ ] Q5 The April Bet: Month-over-month trend analysis (Jan→Feb→Mar), projection for April
- [ ] Q6 Airport vs Non-Airport: Service zone comparison (airports vs boro zones vs yellow zones)
- [ ] Q7 Weekend vs Weekday: Day-of-week breakdowns for demand, revenue, and zone patterns

### Report Page
- [ ] `pages/report.md` contains executive summary, key findings, recommendations, risks, and Plan B
- [ ] All findings reference specific query results or chart data
- [ ] No unsupported claims — every recommendation has evidence

### Slide Deck
- [ ] `reports/swiftride-nyc-playbook/slides/index.html` exists and is self-contained (inline CSS/JS)
- [ ] Fixed 1920×1080 `.deck-stage` with `.slide` class on each slide
- [ ] Keyboard navigation (arrow keys) and button navigation work
- [ ] Only active slide is visible at any time
- [ ] Slide titles make claims (not generic labels)
- [ ] Evidence traceability: each evidence slide references source query/chart
- [ ] Recommendation slides include: action, why now, expected benefit, caveat, next owner
- [ ] Risk/caveat section is prominent, not hidden
- [ ] Plan B / alternative scenario section exists
- [ ] Asset paths are relative (no absolute filesystem paths)
- [ ] `reports/swiftride-nyc-playbook/slides/story.json` exists with story model
- [ ] `reports/swiftride-nyc-playbook/slides/storyboard.md` exists with slide-by-slide outline

### Storyboard Requirements
- [ ] Audience assumption stated (Maya Chen, skeptical, wants actionable decisions)
- [ ] Density mode chosen (Speaker-led recommended for live presentation)
- [ ] Core business question articulated
- [ ] One-sentence answer provided
- [ ] Each slide has: evidence needed, recommended visual, caveats
- [ ] Style recommendation included

## 4. Regression Constraints

- Do not modify `sources/` files (auto-generated by data refresh)
- Do not modify `data/` parquet files
- Do not change `.cmux/workspace.json` intention fields (only update status/pages if needed)
- Do not delete or break existing pages (index.md, draft.md)
- Do not introduce absolute filesystem paths in deck
- Do not create queries that bypass Evidence SQL validation
- Do not use chart types not supported by Evidence (e.g., no custom D3)
- Preserve all existing `queries/` files — only add new ones

## 5. Scope and Boundaries

### In Scope
- `pages/q1.md` through `pages/q7.md` — analysis pages with queries and charts
- `pages/report.md` — polished synthesis report
- `pages/draft.md` — may be used for scratch work
- `queries/` — new SQL files for analysis
- `reports/swiftride-nyc-playbook/slides/` — storyboard, story.json, index.html, assets/
- `sources/files/` — read-only, used for SQL validation

### Out of Scope
- New data files or data refresh
- Changes to Evidence configuration or extensions
- Deployment to production
- PowerPoint export (HTML deck only)
- Mobile-responsive design (1920×1080 fixed stage)
- Interactive filters in the deck (static slides)

## 6. Ambiguities and Required Decisions

### Safe Assumptions
- **Density mode:** Speaker-led (Maya will present live, not read async)
- **Deck pattern:** Decision Memo (she needs to make a deployment decision)
- **Style:** Clean, minimal, data-forward — no decorative elements
- **Slide count:** 15–25 slides (7 question findings + intro + recommendations + risks + Plan B + appendix)
- **Data cleaning:** Filter trips with fare_amount <= 0, trip_distance > 100 miles, or passenger_count = 0/null

### True Blockers
- None identified — all data, tools, and patterns are available

## 7. Blocked Stop Condition

**Stop without marking complete if:**
- Evidence SQL validation fails on any query and cannot be fixed
- `check_evidence_health` reports build errors on any page
- Deck HTML fails to render in browser (broken CSS/JS)
- Story.json cannot be generated (report.md is empty or narrative is incoherent)
- More than 2 of the 7 question pages remain as unexplored placeholders
- Slide quality rubric check produces "Needs story revision" or "Needs technical fixes"
